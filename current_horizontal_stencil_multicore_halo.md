# Current HORIZONTAL_STENCIL Multi-core Halo Plan

## 1. Scope

This document describes the multi-core data partitioning and halo model for the current working LWT device path only:

```text
HORIZONTAL_STENCIL + base accumulation + scale
```

It does not describe future `SHIFT`, `RECOVER1`, `RECOVER2`, or SpliceChain-level propagation.
The goal is to make the current row-major route execution parallel across multiple Tensix cores while preserving the same mathematical result as the existing single-core implementation.

The current implementation has three relevant kernels:

```text
reader  -> packs source/base row-major streams into tiles
compute -> runs HORIZONTAL_STENCIL and adds base
writer  -> writes produced output groups back to row-major DRAM streams
```

The parallelization target is:

```text
one TT-Metal Program
same reader / compute / writer kernels placed on many cores
each core receives a different output-group range
each core reads all source values required for its output range
neighboring cores may duplicate small source reads
each core writes only its owned output range
no overlapping output writes
```

---

## 2. Current Route Model

For every predict/update route, the host planner provides the reader/writer/compute path with route-local geometry:

```text
sourceLength
baseLength
outputLength
outputGroupCount
sourceOffset
baseOffset
sourceLeftPad
kernel width k
```

The route computes one output stream from one source stream and one base stream:

```text
output = base + HORIZONTAL_STENCIL(source, kernel)
```

The dense mathematical equation for the current route is:

$$
y[p]
=
base[baseOffset + p]
+
\sum_{j=0}^{k-1}
h[j]\cdot source[sourceOffset + p + k - 1 - j],
\qquad
p \in [0, outputLength).
$$

This equation is the source of truth for the multi-core halo.

Important: this is a route-local equation. The indices `p`, `sourceOffset + ...`, and `baseOffset + ...` are indices inside the current dense row-major route streams.

---

## 3. What Is Being Parallelized

The work is partitioned across the route output interval:

$$
[0, outputLength).
$$

Each core owns a disjoint output interval:

$$
I_c = [a_c, b_c).
$$

The core must compute exactly:

$$
y[p] \quad \text{for all } p \in [a_c, b_c).
$$

The owned intervals must cover the route output exactly:

$$
[0, outputLength)
=
\bigcup_c [a_c, b_c),
$$

and must not overlap:

$$
[a_c, b_c) \cap [a_d, b_d) = \emptyset
\qquad
\text{for } c \ne d.
$$

Neighboring cores may read overlapping input/source data. They must not write overlapping output data.

---

## 4. Natural Work Unit: Output Group

The current tile path already processes output in groups.

The current constants are:

```text
rows per group        = 32
output blocks per row = 3
block elements        = 16
```

Therefore one logical output group contains:

$$
G = 32 \cdot 3 \cdot 16 = 1536
$$

output samples.

This makes `output_group` the natural unit for the first multi-core implementation.

For a core assigned a group interval:

```text
groupBegin_c
groupCount_c
```

the owned logical output interval is:

$$
a_c = groupBegin_c \cdot G,
$$

$$
b_c = \min((groupBegin_c + groupCount_c)\cdot G,\; outputLength).
$$

The compute kernel can loop over `localGroupCount`.
The reader and writer must use the global group id:

$$
globalGroup = groupBegin_c + localGroup.
$$

This keeps the existing tile layout and writer layout unchanged.

---

## 5. Exact Base Interval Required by a Core

The base term in the route equation is elementwise:

$$
base[baseOffset + p].
$$

For a core-owned output interval:

$$
p \in [a_c, b_c),
$$

the required base indices are:

$$
B_c
=
[baseOffset + a_c,\; baseOffset + b_c).
$$

Therefore the base stream needs no stencil halo. It only needs the same interval length as the owned output interval.

In practice, the reader should treat out-of-range base indices as zero only if required by the existing route boundary behavior. For normal valid route intervals generated by the planner, the base interval should already lie inside the valid base stream overlap.

---

## 6. Exact Source Interval Required by a Core

The source term in the route equation is:

$$
source[sourceOffset + p + k - 1 - j],
\qquad
j \in [0,k).
$$

For one fixed output index `p`, the source indices used are:

$$
sourceOffset + p,
\;
sourceOffset + p + 1,
\;
\dots,
\;
sourceOffset + p + k - 1.
$$

For all output indices:

$$
p \in [a_c, b_c),
$$

the union of required source indices is:

$$
S_c
=
[sourceOffset + a_c,\; sourceOffset + b_c + k - 1).
$$

Proof:

The smallest source index appears at:

$$
p = a_c,\quad j = k - 1,
$$

so:

$$
sourceOffset + a_c + k - 1 - (k - 1)
=
sourceOffset + a_c.
$$

The largest source index appears at:

$$
p = b_c - 1,\quad j = 0,
$$

so:

$$
sourceOffset + (b_c - 1) + k - 1
=
sourceOffset + b_c + k - 2.
$$

In half-open interval form, the required interval is therefore:

$$
[sourceOffset + a_c,\; sourceOffset + b_c + k - 1).
$$

Thus, relative to the owned output interval, the mathematical source halo for the current route is:

$$
H_L = 0,
$$

$$
H_R = k - 1.
$$

So the current route needs a right-side source halo of `k - 1` logical samples.

---

## 7. Why the Halo Is on the Right in the Current Route Formula

The SFPU stencil itself is written as a causal stencil:

$$
g[i] = \sum_{j=0}^{k-1} h[j] f[i-j].
$$

However, the current route equation after planner offsets and reader packing is:

$$
y[p]
=
base[baseOffset + p]
+
\sum_{j=0}^{k-1}
h[j]\cdot source[sourceOffset + p + k - 1 - j].
$$

In this dense route coordinate system, output `p` reads source samples from:

$$
sourceOffset + p
$$

through:

$$
sourceOffset + p + k - 1.
$$

Therefore, when splitting the route output interval across cores, the extra mathematical source samples are on the right side of the core-owned output interval.

This is not a contradiction. It is a result of the current valid-convolution alignment and the existing `sourceOffset` route geometry.

---

## 8. Three Different Padding/Halo Quantities

The current code has three different values that must not be confused.

| Quantity | Formula | Role |
|---|---:|---|
| global wavelet padding | $tapSize - 1$ | physical boundary extension before even/odd split |
| hardware stencil padding | $17 - k$ | SFPU packing/warmup offset used by the reader |
| parallel route halo | $k - 1$ | extra source samples needed by a core for its owned output interval |

Only the last one is the multi-core halo.

### 8.1 Global wavelet padding

Before splitting the original signal into even and odd streams, the preprocessing path pads the original signal by:

$$
P_{wavelet} = tapSize - 1
$$

on both sides.

This is part of the wavelet boundary extension. It is not the per-core route halo.

### 8.2 Hardware stencil padding

The current SFPU stencil supports up to 17 coefficients. For a route kernel width `k`, the reader uses:

$$
P_{sfpu} = 17 - k.
$$

This value appears as:

```text
sourceLeftPad = 17 - k
```

This is a technical packing offset. It exists so the fixed-width tile stencil produces the correctly aligned dense output.

It is not the same thing as the route dependency halo.

### 8.3 Parallel route halo

For the current route equation, if a core owns:

$$
[a_c, b_c),
$$

then it must read source through:

$$
[sourceOffset + a_c,\; sourceOffset + b_c + k - 1).
$$

The additional source region is:

$$
[sourceOffset + b_c,\; sourceOffset + b_c + k - 1).
$$

This is the duplicated read region shared with the next neighboring core.

---

## 9. Actual Reader Source Range with `sourceLeftPad`

The previous section describes the mathematical dependency. The current reader also applies the hardware packing offset:

```text
source_logical_index = packed_index - sourceLeftPad
source_index = sourceOffset + source_logical_index
```

For one output group, the reader builds two input tiles. Because the group produces 3 output half-blocks per row, but the stencil needs the next half-block as input, the packed source region for one group covers one extra 16-sample half-block.

For a core assigned a contiguous output interval:

$$
[a_c, b_c),
$$

aligned to whole output groups, the practical logical source range touched by the current tile packer is approximately:

$$
S^{reader}_c
=
[sourceOffset + a_c - sourceLeftPad,\; sourceOffset + b_c + k - 1).
$$

Since:

$$
sourceLeftPad = 17 - k,
$$

this can also be written as:

$$
S^{reader}_c
=
[sourceOffset + a_c - (17-k),\; sourceOffset + b_c + k - 1).
$$

Interpretation:

- the right side `k - 1` is the mathematical route halo;
- the left side `17 - k` is only the SFPU packing warmup;
- both may be read by the reader;
- only the route output interval `[a_c,b_c)` is written.

For boundary groups, indices outside the valid source stream are handled by the existing zero/out-of-range behavior.

---

## 10. Neighboring Core Overlap

Suppose two neighboring cores own adjacent output intervals:

$$
I_0 = [a, b),
$$

$$
I_1 = [b, c).
$$

Their mathematical source intervals are:

$$
S_0 =
[sourceOffset + a,\; sourceOffset + b + k - 1),
$$

$$
S_1 =
[sourceOffset + b,\; sourceOffset + c + k - 1).
$$

The overlap is:

$$
S_0 \cap S_1
=
[sourceOffset + b,\; sourceOffset + b + k - 1).
$$

Its length is:

$$
k - 1.
$$

This duplicated source read is intentional. It avoids any need for inter-core halo exchange.

No output overlap exists because core 0 writes only:

$$
[a,b),
$$

and core 1 writes only:

$$
[b,c).
$$

---

## 11. Physical Alignment of the Halo

The mathematical halo is measured in logical scalar samples:

$$
H_R = k - 1.
$$

The implementation may round the physical read range to a convenient memory boundary. For example, with 32-float sticks:

$$
stickStart_c =
\left\lfloor
\frac{sourceOffset + a_c - sourceLeftPad}{32}
\right\rfloor,
$$

$$
stickEnd_c =
\left\lceil
\frac{sourceOffset + b_c + k - 1}{32}
\right\rceil.
$$

Then the reader may fetch whole sticks:

$$
[stickStart_c,\; stickEnd_c).
$$

This can read more than the mathematical minimum, but it keeps the data movement simple and aligned with the current row-major stick layout.

The extra physical reads are allowed because reads are non-destructive. The writer still writes only the owned output interval.

---

## 12. Scale Route

Scale routes do not need stencil halo.

For scale:

$$
y[p] = c \cdot source[p].
$$

If a core owns:

$$
[a_c,b_c),
$$

then it only needs:

$$
source[a_c:b_c].
$$

Therefore:

$$
H_L = 0,
\qquad
H_R = 0.
$$

Scale can use the same output group partitioning as predict/update routes, but without source halo.

---

## 13. Multi-core Runtime Model

The current implementation should use the standard SPMD shape:

```text
same kernels
same program
different runtime args per core
```

Each participating core gets route-local work information:

```text
globalGroupBegin
localGroupCount
```

For every route, the host computes:

```text
outputGroupCount = ceil(outputLength / 1536)
```

and splits the group range across participating cores.

Each core uses:

```text
for localGroup in [0, localGroupCount):
    globalGroup = globalGroupBegin + localGroup
```

The reader uses `globalGroup` to pack the correct source/base tiles.

The compute kernel only needs to know how many local groups to compute.

The writer uses `globalGroup` to write the correct output positions.

---

## 14. Required Runtime Arguments

The exact argument layout may change, but conceptually each core needs:

```text
route_config_addr
route_count

for each executable route:
    global_group_begin
    local_group_count
```

The route config already contains route-global information:

```text
source_addr
source_length
base_addr
base_length
output_addr
output_length
output_group_count
source_offset
base_offset
source_left_pad
```

The new per-core information should not replace the route config. It only restricts which part of each route this core owns.

A useful conceptual split is:

```text
route config:
    what this route means globally

per-core runtime args:
    which output groups this core owns
```

---

## 15. Reader Behavior

For predict/update routes, the reader should:

1. load route config;
2. determine `globalGroupBegin` and `localGroupCount` for this core;
3. for each local group:
   - compute the corresponding global group id;
   - pack source tile 0 and source tile 1 using the global group id;
   - pack base tiles using the same global group id;
   - push the tiles into the local CBs.

The key rule:

```text
reader must use global group id for addressing
```

not local group id.

Otherwise every core would read and compute group 0, group 1, etc. instead of its assigned global range.

---

## 16. Compute Behavior

The compute kernel should remain mostly local.

For predict/update:

```text
for group in [0, localGroupCount):
    consume 2 source tiles
    consume 2 base tiles
    run hstencil_plus_base
    produce 2 output tiles
```

The compute kernel does not need to know the global group id if the reader and writer already handle global addressing.

For scale:

```text
for group in [0, localGroupCount):
    consume input tiles
    multiply by scalar
    produce output tiles
```

---

## 17. Writer Behavior

The writer must write only the assigned global output groups.

For every local group:

```text
globalGroup = globalGroupBegin + localGroup
groupBase = globalGroup * 1536
```

The writer writes only values whose output index lies inside:

$$
[0, outputLength).
$$

It must not write halo values. The halo is only used by the reader/compute side to make boundary output values correct.

---

## 18. Correctness Invariant

For every route and every core, the multi-core result must satisfy:

$$
y_{multi}[p] = y_{single}[p]
\qquad
\forall p \in [a_c,b_c).
$$

Since the owned intervals form a disjoint cover of the output interval:

$$
[0, outputLength)
=
\bigcup_c [a_c,b_c),
$$

the full multi-core route output must equal the single-core route output:

$$
y_{multi}[0:outputLength]
=
y_{single}[0:outputLength].
$$

The duplicated source reads do not change values because all cores read from the same immutable source stream for that route.

---

## 19. Validation Plan

The validation should compare the current single-core path against the new multi-core path.

For each tested wavelet scheme and signal length:

1. run the existing single-core implementation;
2. run the multi-core implementation with the same route config;
3. compare final even and odd streams;
4. optionally compare route-by-route debug outputs;
5. verify that only owned output intervals are written by each core.

The route-local invariant is:

$$
\forall c,\ \forall p \in [a_c,b_c):
\quad
y_c[p] =
base[baseOffset+p]
+
\sum_{j=0}^{k-1}
h[j]\cdot source[sourceOffset+p+k-1-j].
$$

The expected numerical tolerance should match the existing single-core vs Python tolerance because the arithmetic order inside each output element does not change.

---

## 20. Summary

For the current implementation, the multi-core halo is simple and route-local.

For a predict/update route:

$$
y[p]
=
base[baseOffset + p]
+
\sum_{j=0}^{k-1}
h[j]\cdot source[sourceOffset+p+k-1-j].
$$

If a core owns:

$$
[a_c,b_c),
$$

then it needs:

$$
B_c =
[baseOffset+a_c,\; baseOffset+b_c)
$$

and:

$$
S_c =
[sourceOffset+a_c,\; sourceOffset+b_c+k-1).
$$

Therefore the mathematical source halo is:

$$
\boxed{H_R = k - 1}
$$

and:

$$
\boxed{H_L = 0}.
$$

The current reader may also read:

$$
17-k
$$

extra logical samples on the left because of SFPU tile packing. This is hardware padding, not mathematical route halo.

The implementation strategy is:

```text
partition by output groups
use global group id in reader/writer
compute local group count
read duplicated source halo
write only owned output
no inter-core halo exchange
```
